How We Build

From "I have an idea" to "I shipped something I trust" — in six steps.

Building real tools isn't about typing fast. It's about thinking clearly before you build, so you don't end up deep in a half-built mess wondering if it'll hold up. These steps force that thinking at the cheapest possible moment. Complexity is the enemy — if you can't explain it simply, it's probably too complicated.
↓ Click any step. We'll follow one real example the whole way: a simple expense tracker — it imports your bank transactions, sorts them into categories, and shows you a monthly report.

Why this exists

Every build needs a place to live, with a little structure so you (and Claude) can always find what's what. This runs once, up front. You don't need to know the structure — the skill figures it out from a short conversation and sets it up for you.

Gate 1/memoDecide if it's even worth buildingshould this exist?▶

The question this gate forces

What problem does this solve, and is the payoff worth the effort? What is it in one sentence — and just as important, what is it NOT? (That boundary is what stops a small idea from sprawling into a monster.) And: how will you know it worked?

This is what & why only — never how. You don't know the "how" yet, and pretending you do is the #1 way people get in over their heads.

Our example

Problem
I never know where my money goes; categorizing transactions by hand takes an hour a month and I stop doing it.
It is
A tool that auto-imports and categorizes transactions into a monthly report. It is NOT budgeting, forecasting, or bill-paying.
Success looks like
At month-end I open one report and see spend by category, with no manual sorting.

✓ If you can't write that one-sentence problem, you've failed the gate yourself — and saved a build that wasn't ready.

Gate 2/prdDesign it properly before buildingis it designed right?▶

The question this gate forces

Can you draw how it works? The skill maps out the pieces and how they connect, checks that everything you're assuming is actually true, and reuses what already exists instead of rebuilding. The goal: by the end, the build is so well-understood it can be done in one shot with zero surprises.

If you can't explain it in a short paragraph, the design isn't clear yet — that's the gate doing its job.

Our example

The pieces (drawn simply)
Bank Import → Categorizer → Reports. The Import pulls transactions; the Categorizer tags each one; Reports add them up by category.
Checked, not assumed
Does the bank actually let us pull transactions? Tested live before committing — yes, with an API key.
Reused, not rebuilt
No custom database — a single spreadsheet/table holds transactions.

✓ Design verified end-to-end. Nothing left to "figure out during the build."

Execution/buildMake exactly what you designedno new decisions▶

Why this exists

Building is pure execution — the thinking already happened in the design. If a real decision suddenly surfaces here, that's a signal the design wasn't finished, and the skill sends you back rather than letting you wing it. That discipline is what keeps builds from quietly going off the rails.

Our example

What happens
Claude builds the import, the categorizer, and the report — checking each piece against the design as it goes, and keeping a log of what it made.

Then: a quiet but critical check

When the build finishes, it checks one thing — does this cross the line into "other people / real money / real data"? Our expense tracker touches your real bank data and real dollar amounts. So it does cross the line — and the build stops and hands you to the final gate. It will not "go live" on its own.

✓ Built to spec — and correctly refused to ship itself, because the stakes are real.

Gate 3/shipMake sure it's safe to rely ononly if the stakes are real▶

The question this gate forces

This gate only fires when the blast radius is real — someone else depends on it, it changes real data, its output drives decisions, or it touches money. When it fires, you answer: what happens when it breaks? Who notices? What's the fallback? Every weak spot gets a plan before anything goes live.

Our example

What breaks?
The bank import could double-count a transaction → your totals are wrong.
Who notices?
Right now, nobody — a wrong total looks like a real total. That's the danger.
The fallback
Add a duplicate check on import; show the raw transaction count so a wrong number is visible. Now it's safe to rely on.

✓ Every hole has a plan. Now it goes live.

Anytime/quick-fixSmall thing broke, or a tiny tweakno ceremony▶

Why this exists

Not everything is a whole build. A typo, a wrong value, a small bug — `/quick-fix` understands the problem deeply, fixes the smallest thing, verifies it, and stops. If it turns out to be bigger than expected, it tells you to start with `/memo` instead.

Example
/quick-fix the report is grouping by week instead of month

The three questions every good build answers

What breaks?Every tool fails somehow. Gate 3 makes you name how, before it bites you.
Who knows?A silent failure is the worst kind. If something goes wrong, someone has to find out.
What's the fallback?When it breaks, is there a way forward? A plan beats a panic.
